# Maple v0.3 Release Gates Workflow (P3-T10)

Status: Active  
Date: 2026-03-27  
Owner: Codex

## 1. Purpose

Define a repeatable, auditable workflow for RC gate evaluation and KPI trend reporting.

This workflow is executable through:

- `tools/release/generate_release_kpi_report.py`

The script emits machine-readable statuses:

- `pass`
- `warn`
- `fail`

## 2. Gate and KPI Binding

The workflow binds each gate to deterministic evidence commands.

| Gate | Blocking Criteria | Primary Evidence Commands |
|---|---|---|
| `G1` build+baseline | configure/build and deterministic baseline must pass | `cmake -S . -B build`, `cmake --build build --config Debug`, `ctest -R maple_tests_unit`, `ctest -R maple_tests_integration`, `ctest -R maple_guard_vm_only` |
| `G2` conformance+diagnostics | contract suites must pass | `ctest -R maple_tests_conformance`, `ctest -R maple_tests_diagnostics` |
| `G3` diagnostics precision | precision threshold assertions must pass | `ctest -R maple_tests_diagnostics` |
| `G4` GC stability | deterministic robustness profile and GC metrics must be valid | `maple_tests --suite=robustness` |
| `G5` robustness critical policy | robustness checks + selected stress scripts pass; S0/S1 backlog must be zero | `maple_tests --suite=robustness`, `build/.../maple_cli tests/scripts/migration/m5_gc_stress_mix.ms`, `build/.../maple_cli tests/scripts/migration/m6_vm_only_convergence.ms` |
| `G6` docs+traceability | release/report artifacts and tracker evidence must exist | generated report artifacts + tracker/design docs |

KPI thresholds follow `docs/design/release-gates-kpi-v0.3.md`.

## 3. Command Sequence

Run the report generator from repository root:

```powershell
python tools/release/generate_release_kpi_report.py --release v0.3-rc1 --config Debug --s0-open 0 --s1-open 0 --s2-open 0
```

The command executes:

1. `cmake -S . -B build`
2. `cmake --build build --config Debug`
3. `ctest --test-dir build --output-on-failure -C Debug`
4. gate-targeted `ctest -R ...` suites
5. `maple_tests --suite=migration_debt`
6. `maple_tests --suite=robustness`
7. selected migration stress scripts using `maple_cli`

## 4. Output Artifacts

The script writes both files under `docs/release/reports/`:

1. `<release>_<timestamp>.json` (machine-readable)
2. `<release>_<timestamp>.md` (human-readable RC summary)

Each report includes:

- candidate metadata (`release`, commit, branch, generation time)
- `G1..G6` status snapshot (`pass|warn|fail`)
- KPI table (`value`, `threshold`, `status`, `source`, `owner`)
- command evidence list with exit code and duration
- RC decision (`promote` or `hold`)

## 5. RC Decision Rule

1. `promote`: all gates are `pass`.
2. `hold`: any gate is `fail` or `warn`.

If waivers are used, track them in release records and convert affected `warn` gates back to `pass` only after policy approval.
